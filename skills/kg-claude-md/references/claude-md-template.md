# CLAUDE.md Output Template

This file defines the required structure, section purposes, and content guidelines for
the CLAUDE.md generated by the kg-claude-md skill.

The CLAUDE.md is written for Claude (or any AI assistant) reading it in future sessions
to understand the vault and act correctly within it. It is also useful for the human owner
as a reference document.

Write it in second person directed at Claude: "When adding a note...", "Always use...", "Check for..."

### 9. Extending the Ontology

This section is critical — it must be prescriptive about when and how to grow the schema.

```markdown
## Extending the Ontology

The ontology should grow when the existing types and domains genuinely don't cover a
new kind of information — not just because a new topic appears. A new topic in an
existing type (e.g., a new project in a new domain) does not require a new type.

### When to add a new entity type

Add a new `type:` value when:
- You have 3+ notes that fundamentally describe the same *kind of thing*
- That thing requires frontmatter fields that don't fit existing types
- The existing types would misrepresent what the note is about

Do NOT add a new type when:
- You just want a different folder or tag for organizational convenience
- One or two notes could fit into an existing type with minor adaptation
- The distinction is only about domain, not about the nature of the entity

**Process for adding a new type**:
1. Write 3 example notes using the new type's schema
2. Verify they can't reasonably fit existing types
3. Define: required fields, optional fields, body structure, linking patterns
4. Add the type to this CLAUDE.md under Entity Types
5. Add it to the `type:` field vocabulary in templates if templates exist

### When to add a new domain

Add a new domain when:
- You're capturing notes on a topic area that doesn't fit any existing domain
- You expect at least 5+ notes in this area over time

**Process**: Add to the Domain Taxonomy table with a brief description. Use it immediately.

### When to add a new tag group

Add a new tag prefix group (e.g., `#context/`) when:
- You need to filter notes by a cross-cutting property not covered by existing groups
- The property is meaningfully different from type, domain, or status

Avoid: tags that are just another way to express what `type:` already captures.

### When to add new frontmatter fields

Add a new field when:
- You find yourself wanting to Dataview-query on information not currently in frontmatter
- The field would apply to multiple notes of the same type (not just one)

Add it to the relevant type's schema definition in this file when you do.

### Deprecating old patterns

When retiring a type, domain, or tag:
- Do not delete existing notes using it
- Mark the old value as deprecated in this file with a migration path
- Migrate notes opportunistically (when editing for other reasons), not in bulk

Example:
```
### `old-type` ⚠️ DEPRECATED
Superseded by `new-type` as of YYYY-MM-DD. 
New notes: use `new-type` instead.
Existing notes: migrate when editing.
```
```

### 10. Common Operations

```markdown
## Common Operations

### Adding a new note

1. Determine the `type` from the Entity Types section
2. Determine the `domain`
3. Create the file at `type/kebab-case-title.md`
4. Fill required frontmatter fields
5. Write a one-sentence summary as the first line of the body
6. Add at least 2 outgoing wikilinks
7. Check whether any existing notes should link *to* this new note, and add those links

### Processing inbox notes

Notes in `_inbox/` are unprocessed captures. For each one:
1. Determine the correct type
2. Move to the appropriate folder with the correct name
3. Fill required frontmatter
4. Add wikilinks
5. Delete from inbox (the promoted note replaces it)

### Updating an existing note

- Update the `updated:` date
- If status has changed, update `status:`
- If the note resolves a problem or produces a decision, update relationship arrays on both notes

### Archiving a note

- Move to `archive/` folder (or set `status: archived` if no archive folder)
- Do not delete — archived notes maintain graph integrity

### Auditing for orphans

Notes with no outgoing links are graph dead-ends. Periodically:
- Look for notes with no `[[` in their body and no relationship arrays
- Add at least one substantive outgoing link, or move to inbox for review
```

## Content Guidelines

**Tone**: Direct and imperative. "Use X format", "Always include Y", "Do not Z".
Not: "You might want to consider using X format."

**Examples**: Every entity type section must include a concrete YAML example with real-looking
values drawn from the vault itself (or credibly extrapolated from it). Avoid abstract placeholders.

**Accuracy over completeness**: If a pattern isn't clearly established in the vault, say so
explicitly: "This field is used inconsistently — use [recommended format] for new notes."
Do not invent standards that aren't grounded in what's actually there.

**Vault-specific language**: Use the actual domain names, type values, and tag prefixes found
in the vault. Do not use generic examples from outside this vault's vocabulary.

**Length calibration**:
- Nascent vault (<50 notes): shorter, more aspirational, focus on establishing patterns
- Developing vault: full detail on established types, briefer on rarely-used ones
- Established vault: comprehensive, include deprecation notes, flag all inconsistencies

---

## What CLAUDE.md Is NOT

- Not a user manual for Obsidian features
- Not a guide to PKM philosophy or methodology
- Not a description of what the vault *should* be (only what it *is*, plus extension rules)
- Not exhaustive API documentation — focus on what Claude needs to act correctly

The file should be actionable on its own. If Claude reads only CLAUDE.md, it should be
able to correctly file any new note into the vault without making structural mistakes.
