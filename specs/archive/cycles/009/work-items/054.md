# WI-054 Implementation Notes: Add automated tests for repair_frontmatter.py

## Context

`scripts/repair_frontmatter.py` was delivered in WI-053 with no test file. The affected-only
test runner (`tests/_dependency_map.py`) only maps modules under `src/`, so any change to the
repair script silently reports "no tests" and exits clean. The gap analyst (cycle 004 MI1)
identified this as a data-safety risk: the script modifies vault files on disk, and regressions
would reach users running `--apply` against live vaults with no automated signal.

## What to Import/Test

Read `scripts/repair_frontmatter.py` and identify the testable functions. Expected:
- `parse_frontmatter(text)` — splits a file into (fm_lines, body) or indicates no frontmatter
- `find_duplicate_keys(fm_lines)` — returns {key: count} for keys appearing more than once
- `deduplicate_frontmatter(fm_lines)` — returns deduplicated list of lines, last-wins
- `repair_file(path, apply)` — orchestrates parse → detect → dedup → optionally write

The function names may differ; read the actual file before writing tests.

## Test Organization

Use pytest with `tmp_path` fixture for file-based tests. No mocks, no LLM calls.

## Test Cases

### parse_frontmatter (or equivalent)

1. Plain markdown with no `---` → returns no-frontmatter indicator
2. Valid frontmatter with body → returns frontmatter lines + body correctly separated
3. `---` in body but valid frontmatter → body content is not mistaken for closing delimiter
   - Example: note with `## Heading\n---\nMore content` in body
4. Opening `---` with no closing `---` (malformed) → handled gracefully (no crash)

### find_duplicate_keys (or equivalent)

1. Frontmatter with no duplicate keys → returns empty result
2. Single key duplicated (`type` appears twice) → correctly identified
3. Multiple keys duplicated → all identified with correct counts
4. Block-sequence continuation lines (`  - python`) → NOT counted as keys

### deduplicate_frontmatter (or equivalent)

1. No duplicates → output equals input
2. Single duplicate key → last occurrence kept, earlier removed
3. Block-sequence value → continuation lines preserved under last occurrence
4. Order: first occurrence of a key establishes position; subsequent occurrences replace it

### repair_file / end-to-end

1. File with no duplicates → not modified (idempotent no-op)
2. File with duplicates, dry_run=True → not written to disk, returns diff description
3. File with duplicates, apply=True → written, content matches deduplicated form
4. Calling repair twice on the same file → second call is a no-op (idempotent)
5. Body after closing `---` preserved exactly, including trailing newlines
