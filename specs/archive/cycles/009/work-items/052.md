# WI-052 Implementation Notes: Fix duplicate frontmatter fields in cb_enrich

## Defect Summary

`_apply_frontmatter_update()` in `src/cyberbrain/mcp/tools/enrich.py` (lines 114–178) appends
new field values as raw strings before the closing `---` without removing existing instances of
those fields first. Each `cb_enrich` invocation with `overwrite=True` appends another copy of
`type`, `summary`, `tags`, and `cb_modified`, leaving the old values in place. YAML parsers
silently take the last value, so the parsed result is correct, but the file on disk accumulates
duplicate keys — breaking Obsidian frontmatter display and any strict YAML consumer.

## Affected Code

- `src/cyberbrain/mcp/tools/enrich.py`
  - `_apply_frontmatter_update()` lines 114–178 — the bug
  - `_format_fm_fields()` lines 181–194 — can be reused

- `tests/test_setup_enrich_tools.py` — add regression tests

## Root Cause

Lines 166–167:
```python
lines_to_add = _format_fm_fields(fields_to_set)
new_content = content[:fm_end] + "\n" + "\n".join(lines_to_add) + content[fm_end:]
```

Inserts new lines before `---` without stripping existing occurrences of the same keys.

## Fix Approach

Rewrite `_apply_frontmatter_update()` to operate on the frontmatter block line-by-line:

1. Extract the frontmatter block (text between opening `---` and closing `---`)
2. Split into lines
3. Build a filtered list: drop any line whose key matches a key in `fields_to_set`
   - Key detection: `line.split(":", 1)[0].strip()` for simple scalar lines
   - Also strip continuation lines for multi-line values of those keys (not needed for
     the fields we manage — type, summary, tags, cb_modified are all single-line)
4. Append the new field lines (from `_format_fm_fields`) at the end of the filtered list
5. Reconstruct: `"---\n" + "\n".join(filtered + new_lines) + "\n---" + body`

This approach:
- Preserves all unmanaged fields exactly as-is (no YAML round-trip reformatting)
- Removes duplicates of managed fields before inserting the new values
- Works for both the normal case (no duplicates) and the repair case (accumulated duplicates)
- Does not introduce a new dependency (no ruamel.yaml needed for this function)

## Keys Managed by enrich (must be deduplicated)

- `type`
- `summary`
- `tags`
- `cb_modified`

All other keys must be preserved verbatim.

## No-Frontmatter Path

The `else` branch (lines 168–172) creates a fresh frontmatter block — no duplication risk there.
Leave it unchanged.

   contains exactly one `type:` line with value `insight`.

2. **Repair of existing duplicates**: Create a note whose frontmatter already contains

   Call `_apply_frontmatter_update`. Assert the result has exactly one `type:` line.

3. **Field preservation**: Call `_apply_frontmatter_update` on a note with `id`, `date`,
   `session_id`, `project`. Assert those fields are unchanged after the update.
