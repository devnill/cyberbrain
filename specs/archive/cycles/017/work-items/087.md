# WI-087: Add search backend cache invalidation on cb_configure

## Problem
`shared.py` has a module-level `_search_backend` global that is never invalidated. When `cb_configure` changes `search_backend` or `embedding_model`, the old backend persists until the MCP server restarts.

## Implementation

### shared.py
Add an invalidation function:
```python
def _invalidate_search_backend():
    global _search_backend
    _search_backend = None
```

### manage.py (cb_configure)
After writing config changes, check if the changed keys include `search_backend` or `embedding_model`. If so, call `_invalidate_search_backend()`.

Look at how cb_configure currently processes config changes — it writes to config.json via `_write_config()` or similar. Add the invalidation call after the write, gated on the changed keys.

### Tests
Add a test in test_manage_tool.py that:
1. Configures a search backend
2. Calls cb_recall (or checks _get_search_backend()) to populate the cache
3. Changes the backend via cb_configure
4. Verifies the cached backend was reset
