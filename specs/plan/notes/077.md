# WI-077: Fix search_backends.py dead branch

## Problem

`search_backends.py` line 862:
```python
if _has_fastembed() or True:  # FTS5 is always available (stdlib sqlite3)
    return FTS5Backend(vault_path, db_path)

return GrepBackend(vault_path)
```

The `or True` makes the condition always true, so `GrepBackend` is unreachable. The comment explains the intent (FTS5 is always available via stdlib sqlite3), but the `or True` was likely a debugging artifact or a shortcut that removed the fallback path.

## Fix

Remove the `or True`. FTS5 availability should be checked properly, or if FTS5 is genuinely always available (sqlite3 is stdlib), then remove the dead `GrepBackend` return and the conditional entirely — just return `FTS5Backend` directly after the hybrid check fails.

The correct logic for auto selection:
1. If hybrid deps available → HybridBackend
2. Else → FTS5Backend (sqlite3 is stdlib, always available)
3. GrepBackend is only selected when `config.search_backend == "grep"` explicitly

If GrepBackend should remain as a fallback for environments where sqlite3 FTS5 extension is somehow unavailable, wrap in a try/except. Otherwise, simplify to just return FTS5Backend.

Look at lines 840-856 for how explicit backend selection works (`hybrid`, `fts5`, `grep` each have their own branch). The `auto` path at 858-865 should follow the same pattern but with fallback logic.
