# WI-088: Refactor cb_extract to share orchestration with main()

## Problem
`cb_extract` in `src/cyberbrain/mcp/tools/extract.py` reimplements parts of the orchestration that `extract_beats.py` `main()` handles (dedup check, runs log, journal writing). This creates risk of behavior divergence.

## Implementation

### extract_beats.py
Extract the core orchestration from `main()` into a new function:
```python
def run_extraction(transcript_path: str, session_id: str, trigger: str, cwd: str, config: dict | None = None) -> dict:
    """Shared extraction orchestration for both CLI and MCP paths.

    Returns a result dict with: beats_count, output_paths, skipped (dedup), etc.
    """
```

This function handles:
1. Config loading (if not provided)
2. Dedup check via run_log
3. Transcript parsing
4. Beat extraction via extractor
5. Vault writing (with autofile if enabled)
6. Search index update
7. Run log entry
8. Journal entry (if configured)

`main()` becomes a thin CLI wrapper that parses args and calls `run_extraction()`.

### extract.py (MCP tool)
Replace the reimplemented orchestration with a call to `run_extraction()`. The MCP tool adds:
- Parameter validation
- Error formatting for MCP response
- Progress reporting

### Tests
Existing tests should continue to pass. The refactor is internal — the external behavior (what main() does, what cb_extract does) stays the same. The key verification is that both paths now produce identical dedup, run log, and journal behavior.

## Risk
This is a medium-complexity refactor touching the core extraction path. The shared function must handle all the edge cases that main() currently handles (missing config, transcript parse failure, backend error, etc.).
