## Significant Findings

### S1: Three import sites still evaluate Path.home() at module import time

The work item converts `state.py` constants to functions to defer `Path.home()` evaluation until call time. However, three modified files immediately call the functions at module scope, recreating the same problem:

- **File**: `src/cyberbrain/extractors/run_log.py:15-16`
  ```python
  EXTRACT_LOG_PATH = extract_log_path()
  RUNS_LOG_PATH = runs_log_path()
  ```

- **File**: `src/cyberbrain/extractors/search_index.py:34`
  ```python
  _SCAN_MARKER_PATH = _index_scan_marker_path()
  ```

- **File**: `src/cyberbrain/extractors/search_backends.py:810`
  ```python
  _DEFAULT_DB_PATH = str(_search_db_path())
  ```

- **File**: `src/cyberbrain/mcp/tools/recall.py:29-30`
  ```python
  _DEFAULT_DB_PATH = str(_search_db_path())
  _DEFAULT_MANIFEST_PATH = str(_search_manifest_path())
  ```

- **File**: `src/cyberbrain/mcp/shared.py:40`
  ```python
  RUNS_LOG_PATH = _runs_log_path()
  ```

That is five module-level calls to state functions, across four files. Combined with `config.py:16` (`GLOBAL_CONFIG_PATH = _config_path_fn()`), that is six module-level evaluations. The WI-093 spec says "all paths computed via functions" and "update all 12 import sites to call the function at use site." These sites call the function at module load, not at use site. `Path.home()` is still evaluated at import time.

**Impact**: Tests that monkeypatch the home directory after import will not affect these cached path values. The patchability argument documented in the code comments (`# patchable in tests via monkeypatch.setattr(...)`) is only true because the tests patch the module-level attribute name, not because the path is resolved lazily. Any new test that patches `Path.home` directly (rather than patching the module attribute by name) will not work for these modules.

**Suggested fix**: The three patterns here are different in intent:

1. `run_log.py` and `shared.py` (EXTRACT_LOG_PATH, RUNS_LOG_PATH): These are tested by patching the module attribute directly, which works. If that pattern is acceptable, the comment should say "evaluated at import time; override via `patch.object(module, 'NAME', ...)`" rather than implying lazy evaluation. If true laziness is required, use the function at call sites: replace `EXTRACT_LOG_PATH` with `extract_log_path()` everywhere it is used inside the module.

2. `search_index.py` (_SCAN_MARKER_PATH): Tests in `test_auto_indexing.py` patch this attribute with `patch.object(si, "_SCAN_MARKER_PATH", marker)`, which works for the existing tests. Same treatment as above applies.

3. `search_backends.py` and `recall.py` (_DEFAULT_DB_PATH, _DEFAULT_MANIFEST_PATH): These are used as fallback defaults in `config.get("search_db_path", _DEFAULT_DB_PATH)`. No tests patch these directly. To make them truly lazy, remove the module-level assignment and call the function inline: `config.get("search_db_path", str(search_db_path()))`.
