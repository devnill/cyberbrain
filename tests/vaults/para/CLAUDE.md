# Vault Guide

This vault uses the PARA methodology (Projects, Areas, Resources, Archive) for organizing knowledge.

## Folder Structure

- **Projects/** — Active projects with defined outcomes and deadlines
- **Areas/** — Ongoing areas of responsibility (no end date)
- **Resources/** — Reference material, patterns, and tools
- **Archive/** — Completed or inactive projects

## Beat Types

| Type | Description |
|---|---|
| `decision` | A choice made between alternatives, with rationale |
| `insight` | A non-obvious understanding or pattern discovered |
| `problem` | Something broken, blocked, or constrained |
| `reference` | A fact, command, snippet, or configuration detail |

## Conventions

- Project notes go in `Projects/<project-name>/`
- Cross-cutting technical references go in `Resources/`
- Operational concerns go in `Areas/`
- When a project completes, move its folder to `Archive/`
- Tags should include the project name for project-scoped notes
- Use wikilinks to connect related notes across PARA categories
